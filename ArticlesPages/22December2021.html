Inversion of Control


30 August 2021 - Written by Mostafa Shaaban Mohamed

Introduction:

Inversion of Control (IoC) is a design principle, used to help make application loose coupling (classes less dependable on each other). Here, the word “Control” means the additional responsibilities assigned to classes that already have their own responsibility. As the name suggests, IoC is about inverting the control. It means to let something else take the control. For example, having a driver to drive your car instead of you doing it yourself so you can focus on something else. This is inverting the control of car from you to driver.

The Flow of Control:

In typical console application in C#, the main method in program.cs control the flow of the application. For example, if we build application to get and print data about the user. The main method will be the responsible for controlling the process (the flow) whether we want to display it on console or GUI.

IoC can also be applied on dependency objects such as classes. Let’s see the code below to understand. 

              
                public class X
                {
                  Y y;

                  public X()
                  {
                    y = new Y();
                  }

                  public void Some_Task() {
	                   y.SomeFunction();
                   }
               }

               public class Y {
                 public void SomeFunction () {
                   //in real world this will do something
                 }
               }

As you can see in the example, Class X needs Class Y to do some task, which means class X is dependent on class Y. We want to change that. IoC means to invert the control. In this case, Class X control the creation of class Y. so we need to invert the control to another class which makes it the only responsibility for this class. Let’s say class Z, so class Z will take control of creating class Y and that will be its only responsibility. While class X can use it to do some task 

              
                public class X
                {
                  Y y;

                  public X()
                  {
                    y = Z.CreateObjectY();
                  }

                  public void Some_Task() {
                    y.SomeFunction ();
                  }
                }

                public class Z
                {
                  public static Y CreateObjectY()
                  {
                    return new Y();
                  }
                }

From the above code, class X uses class Z to create class B which then passes to class A and do some task. Class X is no longer responsible for creation of class Y, it just uses it to do some task.

In OOP design, the classes should not be tightly coupled, it is the opposite it should be design in loosely coupled. It means the change in any class shouldn’t affect the other classes. This way, the application will be maintainable and easy to be tested.

To apply loosely coupled, we should use IoC principle and DIP principle together. Just remember, IoC and DIP are principle which mean they are high level guidelines and doesn’t have a implementation details. So we are free to implement. Some of patterns to implement are factory, abstract factory, dependency injection, template method patterns. In the above code we used factory method pattern to apply IoC.

Conclusion:

This is a simple implementation of IoC and the first step towards achieving loosely coupled design. As mentioned before, we will not achieve complete loosely coupled classes by only using IoC. Along with IoC, we also need to use DIP and DI (Dependency Injection).
